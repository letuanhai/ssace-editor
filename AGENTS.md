# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A Chromium extension (Manifest V3) that enhances SAS Studio 3.8 (a legacy Dojo 1.x web app) by monkey-patching in the page's MAIN world. Two main features: toggling SAS Studio's built-in editor for a modern Ace editor at runtime, and ~25 independent UX fixes (formerly a Tampermonkey userscript, now fully absorbed into the extension — nothing to install separately anymore). Everything works by reverse-engineered runtime patching — there is no build step, bundler, or test suite.

A live SAS Studio instance for testing runs at `http://192.168.1.72/SASStudio/38/`.

## Components

Everything below lives in one extension, all patching the same app. The repo root is the extension root ("Load unpacked" points at it): `manifest.json` sits at the root, all source files (`sw.js`, `relay.js`, `editor-swap.js`, `ss-fixes.js`, `tools-meta.js`, `defaults.js`, `ace-patches.js`, `popup.html`/`popup.js`, `options.html`/`options.js`, and the custom `ace/` files) live under `src/`, and `assets/` (icons) + `lib/` (vendored Ace, kept byte-identical to upstream `ace-builds` — see `src/ace-patches.js`) stay at the root. Consequences of that split: manifest entries and `chrome.scripting` `files:` lists are extension-root-relative (`src/...`), while `options.html`/`options.js` reference the Ace lib page-relatively as `../lib/...`. `./package.sh` zips exactly `manifest.json src assets lib` into `dist/` (gitignored), leaving out the dev-only rest (docs, `test/`, `SASStudio-3.82/`).

- `sw.js` — service worker, three jobs (no `chrome.commands` handler anymore — the editor toggle / native-mouse toggle / browse / command-palette are all in-page `ss-fixes.js` actions with rebindable hotkeys, e.g. editor toggle defaults to `Ctrl+.`, plus the popup's buttons; the only browser command is `_execute_action`, which Chrome itself handles to open the popup, unbound by default). In-page editor toggles post `{ __ssextBadge }` → `relay.js` → a `chrome.runtime.onMessage` listener here that sets the per-tab `ON`/`OFF` toolbar badge (MAIN-world code can't call `chrome.action`; the popup sets its own badge directly):
  1. ss-fixes injection: on every `tabs.onUpdated` for a `/SASStudio/` URL, injects `tools-meta.js` + `ss-fixes.js` into the MAIN world and calls `window.__ssf.init(settings)` with the persisted patch/hotkey settings from `chrome.storage.local`. The same handler pre-injects `editor-swap.js` (+`ace-patches.js`) and seeds `window.__ssExt.libPath`/`userSnippets`/`aceConfig` (the latter via `getAceConfig()`, storage merged over `DEFAULT_ACE_CONFIG`) so the global command-palette / browse hotkeys and any editor created on that page load already have what they need.
  2. Live snippet apply: on `chrome.storage.local`'s `snippets` key changing, pushes the new text into every open SASStudio tab via `window.__ssExt.applySnippets`.
  3. Live ace-config apply: on `chrome.storage.local`'s `aceConfig` key changing (written either by options.html or, via `relay.js`, by the in-page settings panel), merges it over `DEFAULT_ACE_CONFIG` and pushes it into every open SASStudio tab via `window.__ssExt.applyAceConfig`.
- `relay.js` — ISOLATED-world content script (`*://*/SASStudio/*`, `manifest.json`'s `content_scripts`), the only bridge from MAIN-world code to `chrome.storage`: listens for `window.postMessage({ __ssextAceConfig: config })` (source-checked) and writes it to `chrome.storage.local.aceConfig`. Nothing else.
- `editor-swap.js` — defines `AceEditorAdapter` (full reimplementation of the `SAS.Editor` API on top of Ace) and the `window.__ssExt` singleton (`active`, `activate`, `deactivate`, `toggle`, `loadNewAce`, `browse`, `commandPalette`, `applySnippets`, `applyAceConfig`, `libPath`, `aceConfig`). `commandPalette(libPath)` (libPath optional once `ssExt.libPath` is already known — see the global hotkey path below) serializes through the same `_pending` chain as `toggle`/`browse`: it determines the currently-focused Ace editor *before* opening anything (`focusedAceEditor()` — checks `_textViewers` first, then `appDMS.tabs.getAllTabObjects()` code tabs, via each candidate's `.isFocused()`), then `loadNewAce()`s and calls `openCommandPalette(focusedEditor)` with that editor (or `null` if nothing was focused). The adapter's own in-editor `openCommandPalette` command (Alt-Shift-P) calls the same function with `this.aceEditor`. `openCommandPalette` is built on the stock, vendored `ace/ext/prompt` module (`lib/ace/src-noconflict/ext-prompt.js`) rather than a custom prompt module: it builds an entries array — always `SS-Ext: <label>` entries (one per `SSF_TOOLS` action — which now includes `toggleEditor` and `toggleNativeMouse`, so no special-casing) and, only when a focused editor was passed, that editor's own commands (walking `editor.keyBinding.$handlers`, deduped by name, mirroring `prompt.commands`'s `getEditorCommandsByName`, excluding `["insertstring","inserttext","setIndentation","paste"]`) — plus a side-table of runner functions keyed by each entry's `command` field (needed because `prompt.commands`'s `getCompletions` JSON-clones the entries on every filter, which would strip functions), then calls `ace.require("ace/ext/prompt").prompt(focusedEditor || null, "", { name: "commands", selection: [0, Number.MAX_VALUE], onAccept, getPrefix, getCompletions })`, copying `getPrefix`/`getCompletions` filtering from that same preset. A global in-page hotkey (default Alt+Shift+P, rebindable like any other `ss-fixes.js` action — see the `commandPalette` entry in `SSF_TOOLS`) opens the palette even with Ace not activated and nothing focused: `sw.js`'s `tabs.onUpdated` handler pre-injects `editor-swap.js` and seeds `ssExt.libPath`/`ssExt.userSnippets` on every SASStudio page load, so the no-arg `window.__ssExt.commandPalette()` call has what it needs to load the Ace lib on demand. The popup's "Command palette…" button is an alternative entry point, passing an explicit `libPath` (there is no chrome command for it anymore — the only browser command is `_execute_action`, which opens the popup). On first injection it installs one-time dispatcher patches: `SAS.Editor` becomes a function returning either an `AceEditorAdapter` or the original editor depending on `__ssExt.active`, and `DMSEditor.prototype.createCodeEditor` is wrapped the same way (original saved on `__ssExt`, guarded by `_aceReplacementPatched`). `loadNewAce()` loads the newer Ace library from `lib/ace/src-noconflict/` once (its CSS stays attached permanently — SAS Studio renders its own editor DOM via EditorView.js and never references any `.ace_*` class, so the vendored ace's own styles are inert either way). `activate()`/`deactivate()` swap only the `window.ace` global between the app's original bundled Ace and the newer one — that global still matters because runtime `ace.require(...)` calls (Mode.js/SyntaxColorerAdapter.js) resolve against it for tokenization — then walk every open tab converting its editor in place and re-binding `textChanged`/`selectionChanged`/`caretMoved`. Undo history does not survive a toggle in either direction. `browse(kind, libPath, snippetsText)` (used by `window.__ssExt.browse` for browse_ss prompts) only calls `loadNewAce()`, not `activate()` — it works without activating the editor replacement. `applySnippets(text)` registers the user's custom Ace snippets (`ace/snippets` snippetManager, additive over the built-in set) — no-op until the new Ace lib is loaded. `appDMS.createFileView` is also wrapped once (`_createFileViewPatched`): while active, the "View file as text" viewer gets a read-only `AceEditorAdapter` overlay (registry in `__ssExt._textViewers`) laid over the real `SimpleTextarea`, which is kept alive in the widget tree (just visually hidden) so AppDMS's positional load/refresh code (`AppDMS.js:3931`/`3985`, which indexes `pane.getChildren()[1].getChildren()[0]` rather than going through `tabHolder.simpleTextArea`) keeps working untouched; the widget's `.set("value", ...)` is patched to also mirror writes into Ace. The overlay is always editable; a real edit marks the tab dirty (`*name` in the tab title, matching `DMSEditor.applyChangedIndicationToTab`) and enables saving. Save is wired three ways — a toolbar `Save` button (TXT/LOG only), `Ctrl/Cmd+S`, and vim `:w`/`:wq`/`:x` — all POSTing the edited text to the workspace endpoint (mirrors `DMSEditor.saveFile`'s plain-file path); vim `:q`/`:wq`/`:x` also close the tab. The vim ex-commands are registered once on the shared `ace/keyboard/vim` module and resolve the acting `cm.ace` back to either a text-viewer entry or a code-editor tab, so they work in both. `deactivate()` disposes the adapter, clears the dirty marker, unhides the textarea, and restores its original `.set`; viewers opened while inactive are unaffected. (Note: `loadNewAce()` overrides the module URL for `ace/keyboard/vim`/`emacs`/`sublime`/`vscode` to `keybinding-<name>.js` — this build otherwise resolves them to a 404, which had silently disabled non-Ace keyboard handlers entirely.) Persistent editor config: `ssExt.aceConfig` (`{ darkTheme, lightTheme, options, vimrc }`, seeded by `sw.js` and refreshed by `applyAceConfig(config)`) backs `getAceConfig()`, which the adapter constructor reads for its initial theme/`ace.edit()` options, and which `applyConfig(cfg)` (an adapter method) re-applies live (theme via the existing matchMedia machinery — `this._darkTheme`/`_lightTheme` are read live by the matchMedia handler, not captured at construction — plus `aceEditor.setOptions(cfg.options)`). `applyAceConfig(config)` stores it on `ssExt.aceConfig`, calls `applyConfig` on every open adapter (code tabs + `_textViewers`), and (if `config.vimrc` changed since the last apply) reapplies vimrc — see below. `sw.js` calls it on `aceConfig` storage changes. There is no custom preferences panel: Ace's own stock settings menu (`showSettingsMenu`, default `Ctrl-,`/`Cmd-,` command, also surfaced generically in the command palette's per-editor commands list whenever one is focused) is THE settings UI. Loading `ext-settings_menu.js` (in `loadNewAce()`, replacing what used to be a separate `ext-options.js` load — that file bundles its own copy of `OptionPanel`, so eagerly loading it makes the stock command's lazy `config.loadModule("ace/ext/settings_menu", ...)` a no-op, not a second HTTP load) is enough for the menu to work at all; `installSettingsMenuPersistence()` (called once per `loadNewAce()`, guarded by `ssExt._settingsMenuPatched`) additionally patches `ace.require("ace/ext/options").OptionPanel.prototype.setOption` so panel changes stick: after the original runs (which already fires the panel's own `"setOption"` signal), if `option.path` isn't `"theme"`/`"mode"` it updates `ssExt.aceConfig.options[option.path]`, live-applies via `applyAceConfig`, and `window.postMessage({ __ssextAceConfig: cfg }, "*")`s it for `relay.js` to persist (theme is options-page-only, see below; "mode" isn't part of the persisted config). `options.js` installs the identical prototype hook (guarded by its own flag) for the snippet editor's settings menu, writing straight to `chrome.storage.local.aceConfig` instead of postMessage-ing.

Vimrc: `aceConfig.vimrc` is a small subset of vim config lines (`map`/`nmap`/`imap`/`vmap`, `noremap` variants, `unmap` variants — one mapping per line, `"` comments, unsupported lines just `console.warn` and get skipped) applied to the shared `ace/keyboard/vim` module's `Vim.map`/`Vim.noremap`/`Vim.unmap`. `editor-swap.js`'s `applyVimrcLine(Vim, line)` is the parser (duplicated, not shared, in `options.js` as the same function — MAIN-world and options-page code can't share a file); `applyVimrcConfig(text)` (editor-swap.js) and `installVimExCommands()`'s vim-module-loaded callback both drive it, tracking `ssExt._vimrcApplied` (a counter, for test visibility) and `ssExt._vimrcLastText` (to skip a no-op reapply in `applyAceConfig`). Options.js applies the same parser to the snippet editor's vim handler on init and on `aceConfig` storage changes, with no such tracking (nothing tests it there). Removing a line from the vimrc doesn't undo that mapping until the page reloads — `Vim.unmap` isn't run automatically for a diff, only for whatever the current text still asks to unmap.
- `ss-fixes.js` — ~25 independent SAS Studio UX fixes/features (tab management, tree navigation, keyboard shortcuts, clipboard, context menus), split into `ACTIONS` (one-shot commands, e.g. `reloadCurrentFile`) and `PATCHES` (passive monkey-patches applied once at init, e.g. `keepAlive`). Exposes `window.__ssf = { init(settings), run(name) }`; `init()` waits for `.dijitTreeNode`, applies enabled patches, runs any action `setup()`s, and binds hotkeys; `run(name)` invokes a single action on demand (used by the command palette's `SS-Ext:` entries). The dominant pattern for patches is wrap-and-delegate: save the original method (e.g. `tabs.closeTab`), replace it with a wrapper that calls through.
- `tools-meta.js` — shared `SSF_TOOLS` array (`{name, kind, label, title, hotkey}`), one entry per `ss-fixes.js` action/patch. Plain script (no modules) loaded both as an extension-page `<script>` (popup/options) and injected into the page's MAIN world alongside `ss-fixes.js`.
- `defaults.js` — shared `DEFAULT_SAS_SNIPPETS` (the custom snippet text, additive over ace's own built-in SAS snippets) and `DEFAULT_ACE_CONFIG` (`{ darkTheme, lightTheme, options: {...}, vimrc: "" }` — the persisted-config default; storage wins per-key, shallow merge of the top level and of `options`, `vimrc` merged with a `typeof stored.vimrc === "string"` check so a cleared value wins over the default like snippets), loaded via `importScripts()` in `sw.js` and a `<script>` tag in `options.html`.
- `popup.html`/`popup.js` (`action.default_popup`) — the editor toggle, the native-mouse toggle, a "Command palette…" button, and a link to the options page. The individual per-action buttons this popup used to render (one per `SSF_TOOLS` action) now live inside the command palette instead. The palette button files-injects `editor-swap.js` (idempotent) then calls `window.__ssExt.commandPalette(libPath)` on the active tab and closes the popup.
- `options.html`/`options.js` — all configuration: a checkbox per patch (`chrome.storage.local.fixes`), a hotkey table with per-action record/clear (`chrome.storage.local.hotkeys`), an Editor section (dark/light theme selects populated from `ace/ext/themelist`, a "Vim config" textarea for `aceConfig.vimrc` with an explicit Save button — everything else, including the keyboard handler, is configured from Ace's own stock settings menu now and persists through the same prototype hook described above), and an Ace-editor-backed snippet text box (`chrome.storage.local.snippets.sas`, live-applied on Save via the storage-change listener in `sw.js`) that also consumes `aceConfig` (theme pair + `setOptions` + vimrc). Patch/hotkey changes apply on next page reload; snippet and editor-config changes apply immediately. Sets `ace.config.setModuleUrl` overrides for the four keyboard-handler modules (same fix as `editor-swap.js`'s `loadNewAce`) so the snippet editor can actually use them. Listens to `chrome.storage.onChanged` for `aceConfig` (e.g. a change relayed from the stock settings menu) with an `applying` re-entrancy guard so its own writes don't bounce back into itself.
- `lib/ace/` — PRISTINE upstream `ace-builds` 1.43.3 npm package (byte-identical: `diff -rq` against the npm tarball's `src-noconflict/` and `types/` is empty). No local modifications live here anymore — an upgrade is a drop-in file swap. Don't hand-edit anything under here. `ext-prompt.js` is the stock generic-prompt module the command palette is built on; `ext-settings_menu.js` is the stock settings-menu module (bundles its own `OptionPanel`) the persistence hook patches — see `editor-swap.js` above (`ext-options.js` is no longer loaded — `ext-settings_menu.js` bundles an equivalent copy of the same module under the same id).
- `src/ace-patches.js` — `window.__ssExtApplyAcePatches(ace)`, called once per loaded `ace` instance, to reproduce at runtime (each patch independently try/catch'd and idempotency-guarded) the source modifications from the author's ace fork (github.com/letuanhai/ace — commits by letuanhai only; determined by diffing that fork's commits, not by diffing the built output against upstream). It is deliberately minimal — only the fork's actual changes: (1) `ace/layer/decorators` — `$updateDecorators` fully replaced (pristine 1.43.3 body + the fork's two edits) so the scrollbar-overview cursor bar is theme-aware and every selection range is drawn as a half-width bar; (2) vim-aware Esc — `ace/autocomplete`'s `Autocomplete.prototype.commands.Esc` and `ace/snippets`'s (unexported) `TabstopManager.prototype.keyboardHandler`'s Esc both forward to the vim keyboard handler after detaching, only when vim is the active handler (`TabstopManager` isn't exported, so this grabs its prototype via a throwaway editor + snippet insertion; both modules live in `ext-language_tools`, so on pages that don't load it — the options page — the patch skips silently); (3) `ace/ext/modelist` gets a SAS entry pushed onto it (`Mode` isn't exported, so this clones an existing instance's prototype instead of `new Mode(...)`); (4) `ace/ext/statusbar` — `StatusBar.prototype.updateStatus` fully replaced with the fork's `" Line r/total (n selected), Col c "` 1-based format (skipped when `ext-statusbar` isn't loaded). Not reproduced: mouse/gutter/virtual_renderer/theme differences that showed up when diffing our old vendored build against npm 1.43.3 — those were version drift between the fork's base ace and 1.43.3, not the author's modifications, so they revert to pristine. **Call ordering matters**: `loadNewAce()` calls this after loading `ext-language_tools`/`ext-prompt`/`ext-statusbar` (which register the patched modules) but BEFORE `ext-settings_menu.js` — the latter's bundled `ace/ext/options` snapshots `modelist.modes` at load time into the settings-menu Mode dropdown, so the SAS entry must exist first. The options page does the same via `src/ace-seed.js` (a one-line `<script>` calling `__ssExtApplyAcePatches` between `ext-modelist.js` and `ext-settings_menu.js` — a file, not inline, because the extension CSP forbids inline scripts). Both the MAIN-world `AceEditorAdapter` (each SAS editor) and the options-page snippet editor construct an `ace/ext/statusbar` `StatusBar` as a non-intrusive bottom-right overlay (`pointer-events:none`), its font size tracking the configured `fontSize`; the options page loads `ext-statusbar.js` before `ace-seed.js` so the patch applies there too.
- `src/ace/` — the custom (non-upstream) Ace files that used to live under `lib/ace/`: `mode-sas.js`, `snippets-sas.js` (module id `ace/snippets/sas`), `ext-browse_ss.js` + `ext-browse_ss.d.ts` (custom file/tab-browsing extension). `editor-swap.js`'s `loadNewAce()` points `ace.config.setModuleUrl` at this directory for the `ace/mode/sas`/`ace/snippets/sas` module ids (derived from `libPath` by swapping `/lib/ace/src-noconflict` for `/src/ace`) and loads `ext-browse_ss.js` from here instead of `lib/`. Listed under `web_accessible_resources` (`src/ace/*.js`) in `manifest.json` since these load via a page-injected `<script src>`, not `chrome.scripting`.
- The editor toggle is idempotent and repeatable — no page refresh needed to switch back and forth.

## Key SAS Studio integration points

- `window.appDMS` — main app controller; `.tabs` (SASStudioTabs), `.projects`, `.dialogs`, `.sessionId`, `.baseURL`.
- `SAS.Editor` — the editor API surface the adapter must implement (defined in `SASStudio-3.82/resources/js/sas-commons/controls/CodeEditor.js`).
- `dijit.byId(...)` / `dijit.registry` — Dojo widget access (trees are `projects.tree`, `library.tree`, `destination.tree`).
- Beware: some SAS Studio methods are subtly broken by the app itself (e.g. `dijit.Tree._expandNode` is overridden to return undefined; `ss-fixes.js` re-binds the prototype original).

## Development

No build, no Tampermonkey needed anymore. Smoke test: `node test/smoke.js` — launches headless Chromium with the unpacked extension against the live instance and exercises injection, middle-click close (raw CDP input — catches event-suppression bugs synthetic dispatch can't), reopen, the native-mouse toggle, Ace activation/deactivation, the text viewer (mirroring/dirty/save/vim), the command palette (focused/unfocused/global hotkey), and the aceConfig flow (seeding, live apply, settings-menu persistence via relay.js, vimrc); see the file header for `SS_URL`/`CHROME_BIN` env vars and the playwright requirement. To test changes manually: `chrome://extensions/` → reload the unpacked extension (loaded from the repo root) → refresh the SAS Studio page → toggle with `Ctrl+.` (or the popup's toggle button). To build the publishable zip: `./package.sh` → `dist/sas-studio-ext-<version>.zip` (dist/ is gitignored; it packs `manifest.json src assets lib`, so a new runtime file belongs in one of those). All extension logs are prefixed `[SS Ext]`; expect one line per toggle (`activated`/`deactivated ... N tab(s) ...`) plus errors/warnings only.
